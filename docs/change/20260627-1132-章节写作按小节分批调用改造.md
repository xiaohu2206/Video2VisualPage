# 章节写作按小节分批调用改造

## 背景

当前 `09_chapter_write` 按章节调用写作模型。章节过大时，单次模型输入会包含大量镜头卡片，容易出现输入上下文过大、输出被压缩、正文遗漏或引用追溯不稳定的问题。

`08_outline_plan` 已经支持在章节中生成可选 `subsections`，因此本次改造不改变最终章节产物格式，而是在 `09_chapter_write` 内部把有小节的章节拆成小节批次调用模型，再本地组装回原章节。

## 设计原则

- 不引入输入字符数预算。
- 小节是最小输入单位，不因镜头多而继续拆分。
- 单次模型调用只限制镜头总数，不限制小节数量。
- 小节可以合并到同一次模型调用，但输出必须按小节分开。
- 最终仍输出每章一个 `chapter_*.json`，保持 `10_static_render` 的消费契约不变。

## 配置

新增阶段配置：

```json
{
  "chapter_write": {
    "max_shots_per_call": 20
  }
}
```

`max_shots_per_call` 表示一次写作模型调用最多包含多少个镜头。它不控制输入字符数，也不控制一次调用包含多少个小节。

## 09 阶段改造

`09_chapter_write` 调度逻辑调整为：

1. 无 `subsections` 的章节：保持整章调用。
2. 有 `subsections` 的章节：每个小节作为一个写作单元。
3. 按小节顺序贪心打包，相邻小节总镜头数不超过 `max_shots_per_call` 时放入同一批次。
4. 若某个小节自身镜头数超过阈值，不继续拆分，单独调用并写入 `oversized_subsection` warning。
5. 批次模型输出按小节解析，本地按 outline 顺序组装为章节正文。

批次输出建议使用小节标签：

```txt
<SUBSECTION id="chapter_001_sub_001">
<BODY_MARKDOWN>
小节正文
</BODY_MARKDOWN>
<KEY_POINTS>
- 关键点
</KEY_POINTS>
<REFERENCED_SHOTS>
- shot_001
</REFERENCED_SHOTS>
</SUBSECTION>
```

本地组装规则：

- `body_markdown` 由本地插入 `## 小节标题` 并拼接小节正文。
- `referenced_shots` 汇总所有小节引用并按原镜头顺序去重。
- `key_points` 汇总小节关键点并去重。
- `representative_frame` 从章节引用镜头中选择可用帧。
- `subsection_write_batches.jsonl` 记录每个小节批次的 `subsection_ids`、`shot_count`、`status` 和超阈值小节信息。

## 08 阶段配合

`08_outline_plan` 不需要根据写作调用阈值强行切碎小节。小节仍然服务语义结构。

需要补充的是：如果章节明显过大且没有生成 `subsections`，下游只能整章写作，应在结构风险或 warnings 中提示。后续如果频繁出现这类情况，再调整小节触发策略。

## 10 阶段影响

`10_static_render` 继续读取 `chapter_write/chapter_*.json`，无需改变输入契约。

因为 `body_markdown` 最终仍包含 `## 小节标题`，现有 Markdown 渲染路径可以继续工作。

## 11 阶段改造

QA 增加小节级检查：

- 有 `subsections` 的章节，最终 `body_markdown` 应包含所有小节标题。
- 小节标题顺序应与 `outline.json` 一致。
- `referenced_shots` 必须仍属于当前章节。
- `oversized_subsection` 作为非阻断 warning 汇总，不自动重写正文。

## 验收要点

- 小节数量很多但每个小节很短时，可以合并为一次模型调用。
- 小节自身超过 `max_shots_per_call` 时，仍作为单小节调用，不继续拆分。
- 批次输出缺失某个小节时，只重试缺失小节或对应批次，不影响已成功小节。
- 最终 `chapter_*.json` 契约保持不变。
- 单章重跑仍只替换目标章节文件和 `chapters_index.json` 中对应条目。
